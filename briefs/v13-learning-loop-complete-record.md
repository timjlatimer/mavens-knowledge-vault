The previous Learning Loop solved the **inheritance failure** — the problem that subtasks never received Tim's directives. It designed a 4-layer defense system: Fireproof Safe (storage), Village Crier (injection), Checkpoint (verification), Watchtower (drift detection). That design was sound.

Consider what this means:

- **INJECTION_PROTOCOL.md** (361 lines) describes *how* to inject directives into subtasks. It contains templates, examples, and rules. But it does not **execute** the injection. It is a manual that hopes someone reads it and follows the instructions.
- **VERIFICATION_GATE.md** (227 lines) describes *how* to verify subtask outputs against directives. It contains a three-step process, scan templates, and classification rules. But it does not **run** the verification. It is a checklist that hopes someone performs the checks.
- **DRIFT_DETECTION.md** (339 lines) describes *how* to detect drift across five triggers. It contains NEVER-COMPRESS rules, weekly audit templates, and recovery protocols. But it does not **detect** anything. It is a watchtower with no watchman inside.

**The Smelling Salts reframe**: The Constitutional Memory System has the right architecture but the wrong enforcement model. It is a building code that describes how to build a fireproof safe, but the safe itself has not been built. The documentation describes the locks, but the locks are not installed. Any AI agent — including Master Jeeves — can read these files and choose to ignore them. There is no programmatic mechanism that **prevents** non-compliance. The system relies entirely on the good faith of the AI reading the documentation.

This is the same category of failure that created the KEI incident: **hoping** the system does the right thing instead of **making** it structurally impossible to do the wrong thing.

### Move 37: From Documentation-as-Enforcement to Code-as-Enforcement

The Move 37 insight: **The operational protocols (Layers 2-4) must become running code that executes automatically, not documentation that describes what should happen.** The markdown files for INJECTION_PROTOCOL.md, VERIFICATION_GATE.md, and DRIFT_DETECTION.md should remain as human-readable constitutional source documents — Tim's right to read and understand the system is sacred. But the *enforcement* of those protocols must be programmatic.

This is the difference between:
- A speed limit sign (documentation) vs. a speed governor on the engine (code)
- A "wash your hands" sign (documentation) vs. an automatic hand sanitizer that activates when you enter (code)
- A fire code manual (documentation) vs. an automatic sprinkler system (code)

#### Component 1: The Directive Registry (`directive_registry.py`)

The Directive Registry is the programmatic replacement for "read DIRECTIVES.md and figure out what's relevant." It parses the markdown source files into structured Python objects with queryable attributes.

| Validation | None — any text can be written | Schema validation on directive structure before acceptance |

The Registry reads DIRECTIVES.md as its source of truth but maintains an in-memory structured index. When DIRECTIVES.md changes (detected via Git polling or webhook), the Registry rebuilds its index automatically.

### GUARDIAN 1: THE DEVIL'S ADVOCATE (6 Findings)

**DA-1 (CRITICAL): The Chicken-and-Egg Problem — Who Enforces the Enforcer?**

**Resolution**: The enforcement engine must not be optional middleware — it must be the **only path** to subtask execution. The architecture must be designed so that the subtask spawning mechanism itself routes through the engine. This is the difference between a speed limit sign (ignorable) and a physical barrier (structural). **DA-2 (CRITICAL): Markdown-Code Synchronization Drift**

The database is a derived index, not a primary store.

**Integration**: Add `sync_monitor.py` component with webhook listener and periodic polling fallback.

**DA-4 (SERIOUS): SQLite Single-Point-of-Failure**

**Integration**: Add `backup_sync.py` component that mirrors critical database tables to GitHub as structured files in a `/data/` directory.
